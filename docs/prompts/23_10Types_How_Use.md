# 웹 프로토타입 제작을 위한 상세 요구사항 정의서

### 1. 프로젝트 개요

* **프로젝트명 (가칭):** MyRacingHub (마이레이싱허브)
* **프로젝트 목표:**
    1.  산재된 경마 정보(말, 기수, 경주)를 중앙화하고 시각적으로 정리하여 제공한다.
    2.  사용자가 자신의 '관심 말'을 등록하고 관련 정보를 맞춤형으로 제공받는 '개인화된 대시보드'를 구축한다.

### 2. 핵심 사용자 페르소나 및 시나리오

* **페르소나 1: 초보 경마 팬 (김경마)**
    * **특징:** 경마가 흥미롭지만 용어가 어렵고 어떤 말을 골라야 할지 모름.
    * **요구:** "어떤 말이 요즘 잘 나가요?", "A라는 말의 기본 정보(나이, 성별, 승률)가 궁금해요."
    * **핵심 시나리오:**
        1.  사이트에 가입한다.
        2.  이름으로 말을 검색한다.
        3.  말의 상세 정보(프로필, 최근 5경기 성적)를 확인한다.
        4.  마음에 드는 말을 '관심 말'로 등록(Follow)한다.

* **페르소나 2: 데이터 분석형 팬 (박예측)**
    * **특징:** 경마 데이터를 분석하여 경주 결과를 예측하는 것을 즐김.
    * **요구:** "A 말과 B 기수의 궁합(통산 성적)이 어때요?", "이번 주 경주에 나오는 말들의 상세 데이터를 비교하고 싶어요."
    * **핵심 시나리오:**
        1.  메인 대시보드에서 '오늘의 주요 경주'를 확인한다.
        2.  경주 상세 페이지로 이동하여 출전 말 목록과 상세 스펙(부담중량, 레이팅)을 비교한다.
        3.  과거 경주 데이터를 필터링(날짜, 거리)하여 특정 조건의 결과를 확인한다.

### 3. 기능 요구사항 (Functional Requirements)

#### `[FR-USER]` 회원 관련 기능
* **FR-USER-001:** 이메일/비밀번호 기반의 일반 회원가입 기능.
* **FR-USER-002:** 소셜 로그인 기능 (구글, 카카오).
* **FR-USER-003:** 로그인 / 로그아웃 기능 (JWT 기반 토큰 인증).
* **FR-USER-004:** 마이페이지
    * **FR-USER-004a:** 내 정보 수정 (닉네임, 비밀번호 변경).
    * **FR-USER-004b:** 내가 '관심 말'로 등록한 말 목록 조회 및 관리 (삭제).

#### `[FR-DATA]` 데이터 조회 기능
* **FR-DATA-001:** 말(Horse) 상세 페이지
    * **구성:** 이름, 나이, 성별, 마주, 조교사, 최근 10경기 성적 리스트, 통산 승률.
* **FR-DATA-002:** 기수(Jockey) 상세 페이지
    * **구성:** 이름, 사진, 통산 성적(승률, 복승률), 최근 10경기 성적.
* **FR-DATA-003:** 경주(Race) 상세 페이지
    * **구성:** 경주 일시, 장소, 거리, 조건, 출전 말 목록 (말 이름, 기수 이름, 부담중량).
* **FR-DATA-004:** 경주 결과 페이지 (경주 종료 후)
    * **구성:** 순위, 말 이름, 기수 이름, 배당률 정보.

#### `[FR-DASH]` 대시보드 (개인화) 기능
* **FR-DASH-001:** 메인 대시보드 (로그인 사용자)
    * **구성:** 위젯 기반 (UI 디자인팀과 협의 필요).
* **FR-DASH-002 (핵심 기능):** '관심 말' 등록/해제
    * 사용자는 '말 상세 페이지' 또는 '경주 상세 페이지'에서 특정 말을 '관심 말'로 등록(팔로우)할 수 있다.
* **FR-DASH-003 (핵심 기능):** 개인화 대시보드 위젯
    * **위젯 1:** "나의 관심 말" 목록 (현재 상태: 휴식/훈련/출전 예정).
    * **위젯 2:** "관심 말의 다음 경주" (D-Day, 경주 정보 요약).
    * **위젯 3:** "오늘의 주요 경주" (운영자가 선정).

#### `[FR-SEARCH]` 검색 기능
* **FR-SEARCH-001:** 통합 검색 (Header에 위치)
    * 검색 대상: '말', '기수'
    * 검색 방식: 자동완성(Auto-complete) 지원.

### 4. 페이지 구조 (Screen List) 및 핵심 사용자 흐름 (User Flow)

#### 페이지 리스트 (총 8개)
* **P-01:** 랜딩 페이지 (로그인/회원가입 유도)
* **P-02:** 회원가입 (이메일 / 소셜)
* **P-03:** 로그인
* **P-04:** 메인 대시보드 (로그인 후 보이는 첫 화면, 위젯 구성)
* **P-05:** 말 상세 페이지
* **P-06:** 기수 상세 페이지
* **P-07:** 경주 상세/결과 페이지
* **P-08:** 마이페이지 (내 정보 수정 / 관심 말 관리)

#### 핵심 사용자 흐름 (Flow: '관심 말' 등록 및 확인)
1.  `(P-03)` 사용자가 **메인 대시보드**에 접속한다.
2.  `(P-03)` "오늘의 주요 경주" 위젯에서 특정 경주를 클릭한다.
3.  `(P-07)` **경주 상세 페이지**로 이동. 출전 말 목록을 본다.
4.  `(P-07)` 특정 '말 이름'을 클릭한다.
5.  `(P-05)` **말 상세 페이지**로 이동. 말의 성적을 확인한다.
6.  `(P-05)` 마음에 들어서 '관심 말 등록 (★)' 버튼을 클릭한다.
7.  `(P-03)` (며칠 뒤) 사용자가 **메인 대시보드**에 다시 접속한다.
8.  `(P-03)` "관심 말의 다음 경주" 위젯에 (6)에서 등록한 말이 'O월 O일 출전 예정'이라고 표시되는 것을 확인한다.

### 5. 데이터 요구사항 (기초 DB 스키마 제안)

(개발팀이 이 제안을 기반으로 상세 설계를 진행)

* **Users** (회원)
    * `user_id` (PK)
    * `email` (Unique)
    * `password_hash`
    * `nickname`
    * `provider` (가입 경로: 'local', 'google', 'kakao')

* **Horses** (말 정보)
    * `horse_id` (PK)
    * `horse_name`
    * `birth_date`
    * `gender`
    * `owner_name` (마주)
    * `trainer_name` (조교사)
    * `stats_json` (통산 성적 JSON: {total_races: 10, wins: 2, ...})

* **Jockeys** (기수 정보)
    * `jockey_id` (PK)
    * `jockey_name`
    * `stats_json` (통산 성적 JSON)

* **Races** (경주 정보)
    * `race_id` (PK)
    * `race_datetime`
    * `race_location` (서울, 부산 등)
    * `distance` (1200m, 1800m 등)
    * `race_name` (제X회 OOO배 대상경주)

* **Race_Entries** (경주 출전 목록)
    * `entry_id` (PK)
    * `race_id` (FK to Races)
    * `horse_id` (FK to Horses)
    * `jockey_id` (FK to Jockeys)
    * `weight_penalty` (부담중량)
    * `final_rank` (최종 순위, 경주 후 업데이트)
    * `odds` (배당률)

* **User_Follows** (사용자 관심 말) **(핵심 테이블)**
    * `follow_id` (PK)
    * `user_id` (FK to Users)
    * `horse_id` (FK to Horses)
    * *Constraint: (user_id, horse_id)는 Unique해야 함.*

### 6. 비기능적 요구사항 (Non-Functional Requirements)

* **[NFR-PERF] 성능:**
    * 모든 페이지 로딩 시간은 3초 이내여야 한다. (특히 데이터가 많은 대시보드).
    * 검색 자동완성은 500ms 이내 응답해야 한다.
* **[NFR-DATA] 데이터 소스 및 갱신:**
    * **(필수 확인 사항)** 경마 정보(말, 기수, 경주)는 어디서 가져오는가?
    * **가정:** 외부 API (예: KRA(마사회) 제공 API)를 사용한다고 가정.
    * **요구사항:** 경주 결과는 경주 종료 후 10분 이내에, 신규 경주 정보는 1일 1회 Batch로 업데이트되어야 한다.
* **[NFR-SEC] 보안:**
    * 사용자 비밀번호는 반드시 해시(hash)하여 저장한다 (Bcrypt 사용 권장).
    * 모든 웹 트래픽은 HTTPS(SSL)를 사용한다.
* **[NFR-COMP] 호환성:**
    * **PC:** Chrome, MS Edge, Safari (최신 버전 기준)
    * **Mobile:** 모바일 웹 반응형 지원 (iOS Safari, Android Chrome)
